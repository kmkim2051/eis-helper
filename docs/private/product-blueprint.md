프로젝트 컨텍스트: 정보보안기사 AI 기반 주관식 채점 MVP

1. 프로젝트 개요

본 프로젝트는 정보보안기사 실기 대비 주관식 답안 채점 서비스를 목표로 한다.

단순히 ChatGPT/Gemini 같은 범용 AI에게 “이 답안 채점해줘”라고 요청하는 서비스가 아니라, 정보보안기사 시험 특성에 맞춰 문제별 채점 기준, 정답 키워드, 유사 표현, 부분점수 기준을 구조화하고, 이를 기반으로 사용자의 답안을 일관성 있게 평가하는 서비스다.

초기 MVP는 전과목 대응이 아니라 애플리케이션 보안 과목 단일 영역을 대상으로 한다.
SQL Injection, XSS, CSRF, 인증/인가, 세션 보안, 파일 업로드 취약점, 입력값 검증, 출력값 인코딩 등 개발자 친화적이고 주관식 채점 가치가 잘 드러나는 주제를 우선 지원한다.

서비스 홍보 시에는 “아직 특정 과목만 지원한다”는 부족한 표현보다는, 다음과 같은 방향으로 설명한다.

현재는 채점 품질과 피드백 정확도를 높이기 위해 애플리케이션 보안 과목을 우선 지원하며, 서비스 안정화 이후 네트워크 보안, 시스템 보안, 정보보안관리 및 법규 등으로 지원 범위를 단계적으로 확장할 예정이다.

⸻

2. 핵심 목표

이 프로젝트의 핵심 목표는 다음과 같다.

1. 사용자가 정보보안기사 실기형 주관식 답안을 입력하면 예상 점수를 제공한다.
2. 단순 정답/오답이 아니라 부분점수 관점으로 채점한다.
3. 인정된 키워드, 누락된 키워드, 부분 인정 항목을 명확히 보여준다.
4. 사용자의 답안을 시험장에서 점수를 받을 수 있는 “시험형 답안”으로 개선한다.
5. 누적 풀이 이력을 바탕으로 취약 영역과 반복 누락 키워드를 분석한다.
6. AI는 보조 판단과 피드백 생성에 사용하고, 최종 점수 계산은 서버의 채점 기준 기반 로직으로 수행한다.

⸻

3. 이 서비스가 범용 AI 대비 가져야 할 차별점

범용 AI는 개념 설명이나 답안 개선에는 강하지만, 시험 채점 서비스로는 다음 한계가 있다.

* 같은 답안에 대해 채점 결과가 흔들릴 수 있다.
* 시험 기준의 부분점수 체계가 명확하지 않다.
* 채점 근거가 구조화되어 있지 않다.
* 문제별 정답 키워드와 유사 표현을 안정적으로 관리하기 어렵다.
* 사용자의 누적 학습 이력과 취약 키워드를 장기적으로 관리하지 못한다.

따라서 본 서비스는 다음을 장점으로 가져가야 한다.

* 문제별 채점 기준 기반 채점
* 정답 키워드/유사 표현/오답 표현 DB
* 부분점수 채점
* 인정/누락/부분 인정 항목 분리
* 시험형 답안 변환
* 같은 답안에 대한 일관성 있는 결과
* 사용자 풀이 이력 기반 취약점 분석
* 추후 과목 확장 가능한 데이터 구조

핵심 메시지는 다음과 같다.

ChatGPT는 “선생님처럼 설명”해주지만, 이 서비스는 “채점자처럼 점수화하고, 합격 전략까지 관리”해야 한다.

⸻

4. 초기 MVP 범위

초기 MVP는 애플리케이션 보안 과목 단일 영역으로 제한한다.

초기 지원 주제 예시

* SQL Injection
* XSS
* CSRF
* 파일 업로드 취약점
* 인증과 인가
* 세션 하이재킹
* 세션 고정 공격
* 비밀번호 저장 정책
* 입력값 검증
* 출력값 인코딩
* 접근통제 취약점
* 디렉터리 리스팅
* 경로 조작 공격
* 쿠키 보안 속성
* HTTPS/TLS 적용 목적

초기 문제 수

초기에는 20~30문제 정도를 목표로 한다.
문제 수를 무리하게 늘리기보다, 각 문제에 대해 채점 기준과 피드백 품질을 높이는 것이 우선이다.

MVP에서 제공할 핵심 기능

* 문제 목록 조회
* 문제 상세 조회
* 사용자 답안 제출
* 채점 결과 반환
* 예상 점수 표시
* 인정 키워드 표시
* 누락 키워드 표시
* 부분 인정 항목 표시
* 감점 사유 표시
* 시험형 추천 답안 제공
* 풀이 결과 저장
* 간단한 카테고리/키워드별 취약점 분석

⸻

5. MVP에서 제외할 기능

초기 MVP에서는 다음 기능을 우선순위에서 제외한다.

* 전과목 대응
* 완성형 모의고사 시스템
* 랭킹/커뮤니티
* 결제
* 관리자용 대시보드 고도화
* 법규 최신성 자동 추적
* 복잡한 RAG 기반 지식 검색
* 대규모 벡터DB 기반 추천 시스템
* 모바일 앱
* 실시간 채팅 튜터

초기 목표는 “정보보안기사 전체 학습 플랫폼”이 아니라, 애플리케이션 보안 주관식 답안을 안정적으로 채점하는 엔진을 만드는 것이다.

6. 과목 타겟팅 전략

특정 과목 타겟팅을 선택하는 이유

초기 MVP는 전과목 대응보다 특정 과목 타겟팅이 적합하다.

이유는 다음과 같다.

* 채점 기준 데이터를 빠르게 구축할 수 있다.
* 서비스 메시지가 명확하다.
* 채점엔진 품질을 빠르게 검증할 수 있다.
* 문제별 정답 키워드와 유사 표현 관리가 가능하다.
* 포트폴리오 관점에서 “도메인 특화 AI 채점엔진”으로 설명하기 좋다.
* 전과목 대응 시 발생하는 품질 편차를 줄일 수 있다.

애플리케이션 보안을 먼저 선택하는 이유

애플리케이션 보안은 초기 MVP에 가장 적합하다.

이유는 다음과 같다.

* 개발자 입장에서 도메인 이해가 쉽다.
* SQL Injection, XSS, CSRF 등 답안 표현이 다양하다.
* 정답 키워드와 유사 표현 관리 가치가 크다.
* AI 의미판정의 효용이 잘 드러난다.
* 부분점수 채점과 시험형 답안 변환 기능을 보여주기 좋다.
* 백엔드 개발자 포트폴리오와 연결성이 높다.

추후 확장 순서

추천 확장 순서는 다음과 같다.

1. 애플리케이션 보안
2. 네트워크 보안
3. 시스템 보안
4. 정보보안 일반
5. 정보보안관리 및 법규

법규 과목은 최신성, 용어 변경, 법령 근거 관리가 필요하므로 초기 MVP에서는 제외하고 후순위로 둔다.

7. 핵심 도메인 모델

초기 도메인 모델은 다음 개념을 중심으로 설계한다.

Problem

문제 정보를 나타낸다.

주요 필드 예시:

* id
* subject
* category
* questionText
* totalScore
* answerType
* difficulty
* explanation
* createdAt
* updatedAt

subject 예시:

* APPLICATION
* NETWORK
* SYSTEM
* GENERAL
* MANAGEMENT_LAW

초기 데이터는 APPLICATION만 제공하되, 구조적으로는 전과목 확장 가능하게 설계한다.

answerType 예시:

* DEFINITION
* LISTING
* PROCEDURE
* EXPLANATION
* COMPARISON
* MIXED

⸻

ScoringRubric

문제별 채점 기준 묶음이다.

주요 필드 예시:

* id
* problemId
* totalScore
* gradingPolicy
* description

gradingPolicy 예시:

* KEYWORD_BASED
* SEMANTIC_BASED
* HYBRID

초기에는 HYBRID를 기본 방향으로 한다.
단, v1에서는 Rule 기반 키워드 매칭부터 시작하고, 이후 LLM 의미판정을 추가한다.

⸻

ScoringCriterion

실제 채점 기준 항목이다.

주요 필드 예시:

* id
* rubricId
* content
* score
* required
* gradingType
* orderNo

예시:

문제: SQL Injection 대응 방안을 3가지 쓰시오. [6점]

채점 기준:

* PreparedStatement 또는 바인딩 변수 사용: 2점
* 입력값 검증 또는 화이트리스트 검증: 2점
* DB 권한 최소화: 1점
* 에러 메시지 노출 제한: 1점
* WAF 적용: 1점

⸻

CriterionAlias

채점 기준의 유사 표현을 관리한다.

주요 필드 예시:

* id
* criterionId
* alias

예시:

criterion: PreparedStatement 또는 바인딩 변수 사용

aliases:

* PreparedStatement
* prepared statement
* 프리페어드 스테이트먼트
* 바인딩 변수
* 파라미터 바인딩
* Parameterized Query
* SQL 문자열 결합 방지
* 쿼리 파라미터화

⸻

UserAnswer

사용자가 제출한 답안이다.

주요 필드 예시:

* id
* userId
* problemId
* answerText
* score
* totalScore
* submittedAt

초기에는 로그인 없이 익명 사용자 기반으로 구현해도 된다.
다만 풀이 이력 분석을 위해 userId 또는 sessionId 개념은 고려할 수 있다.

⸻

ScoringResult

답안 채점 결과다.

주요 필드 예시:

* id
* userAnswerId
* score
* totalScore
* summary
* createdAt

⸻

ScoringResultDetail

채점 기준별 상세 결과다.

주요 필드 예시:

* id
* scoringResultId
* criterionId
* status
* awardedScore
* reason
* confidence

status 예시:

* MATCHED
* PARTIAL
* MISSING
* NEGATIVE
* UNKNOWN

⸻

8. 채점엔진 설계

채점엔진은 이 프로젝트의 핵심이다.

중요한 원칙은 다음과 같다.

AI에게 최종 점수를 직접 맡기지 않는다.
AI는 “채점 기준 충족 여부 판단”과 “피드백 문장 생성”에 사용하고, 최종 점수 계산은 서버에서 수행한다.

전체 채점 흐름

1. 사용자가 답안을 제출한다.
2. 서버가 Problem과 ScoringRubric을 조회한다.
3. AnswerPreprocessor가 답안을 전처리한다.
4. KeywordMatcher가 정답 키워드와 유사 표현을 매칭한다.
5. Rule 기반으로 명확히 판정 가능한 항목을 먼저 처리한다.
6. 애매한 항목은 LlmCriterionEvaluator가 의미 충족 여부를 판단한다.
7. ScoreCalculator가 채점 기준에 따라 최종 점수를 계산한다.
8. FeedbackGenerator가 채점 결과 기반 피드백을 생성한다.
9. ScoringResult와 ScoringResultDetail을 저장한다.
10. 사용자에게 결과를 반환한다.

⸻

9. 채점엔진 내부 컴포넌트

ScoringEngine

채점 전체 흐름을 오케스트레이션한다.

역할:

* 문제/채점기준 기반 채점 수행
* KeywordMatcher, LLM Evaluator, ScoreCalculator 호출
* 최종 ScoringResult 생성

⸻

AnswerPreprocessor

사용자 답안을 정규화한다.

역할:

* 공백 정리
* 특수문자 정리
* 영문 대소문자 정규화
* 한글/영문 혼용 용어 정리
* 불필요한 조사/문장부호 제거
* 용어 표준화

예시:

* PreparedStatement
* prepared statement
* 프리페어드 스테이트먼트
* 바인딩 변수

이 표현들을 같은 의미군으로 처리할 수 있도록 전처리한다.

⸻

KeywordMatcher

정답 키워드와 유사 표현을 매칭한다.

역할:

* 직접 키워드 매칭
* alias 기반 매칭
* 필수 키워드 누락 확인
* 오답/위험 표현 탐지

초기 MVP에서는 이 컴포넌트가 가장 중요하다.
처음부터 AI 의미판정에 의존하지 말고, Rule 기반 채점이 먼저 동작해야 한다.

⸻

LlmCriterionEvaluator

AI를 사용해 애매한 표현이 채점 기준을 충족하는지 판단한다.

주의사항:

AI에게 “몇 점인가?”를 묻지 않는다.
AI에게는 “이 답안이 특정 채점 기준을 충족하는가?”만 묻는다.

입력 예시:

* 문제
* 채점 기준
* 사용자 답안
* 이미 매칭된 키워드
* 판단해야 할 criterion 목록

출력 예시:

{
"criterionId": 12,
"status": "PARTIAL",
"confidence": 0.78,
"reason": "PreparedStatement를 직접 언급하지 않았지만 SQL 문자열을 직접 결합하지 않고 파라미터 방식으로 처리한다고 설명했으므로 부분 또는 의미상 인정 가능"
}

status는 다음 중 하나로 제한한다.

* MATCHED
* PARTIAL
* MISSING
* UNKNOWN

⸻

ScoreCalculator

점수 계산을 담당한다.

역할:

* MATCHED 항목에 full score 부여
* PARTIAL 항목에 partial score 부여
* MISSING 항목에 0점 부여
* NEGATIVE 항목은 감점 처리
* 총점 초과 방지
* 배점 합산

예시 정책:

* MATCHED: 100%
* PARTIAL: 50%
* MISSING: 0%
* NEGATIVE: 감점

최종 점수 계산은 반드시 서버 로직에서 수행한다.

⸻

FeedbackGenerator

채점 결과를 바탕으로 사용자 피드백을 생성한다.

역할:

* 왜 감점되었는지 설명
* 인정된 키워드 안내
* 누락된 키워드 안내
* 시험형 추천 답안 생성
* 암기 포인트 제공
* 다음 복습 방향 제안

AI는 이 단계에서 적극적으로 활용할 수 있다.
단, 피드백은 ScoringResultDetail에 기반해야 하며, AI가 임의로 새로운 채점 근거를 만들어내면 안 된다.

⸻

10. AI 사용 원칙

AI를 사용해도 되는 부분

* 의미 유사도 판단
* 애매한 답안의 기준 충족 여부 판단
* 피드백 문장 생성
* 시험형 답안 변환
* 누락 키워드 기반 보완 답안 생성
* 사용자가 이해하기 쉬운 해설 생성

AI에게 맡기면 안 되는 부분

* 최종 점수 계산
* 전체 채점 기준 자동 생성
* 법/제도 최신성 판단
* 합격/불합격 단정
* 출처 없는 정답 확정
* 문제 데이터 무검증 생성

핵심 원칙

AI가 판단한다.
서버가 검증한다.
서버가 점수화한다.
AI가 설명한다.

12. 개발 우선순위

Phase 1: 단일 문제 채점 MVP

목표:

한 문제에 대해 사용자가 답안을 제출하면 점수와 피드백을 받을 수 있다.

구현 범위:

* Problem 조회
* Rubric 조회
* 답안 제출
* Rule 기반 키워드 매칭
* 점수 계산
* 채점 결과 반환
* 기본 추천 답안 반환

이 단계에서는 LLM 없이도 동작 가능해야 한다.

⸻

Phase 2: Rule + LLM 하이브리드 채점

목표:

단순 키워드 매칭으로 놓치는 유사 표현을 AI가 보조 판단한다.

구현 범위:

* LLM Client
* LlmCriterionEvaluator
* JSON 응답 파싱
* confidence 처리
* UNKNOWN 상태 처리
* LLM 실패 시 fallback 처리

주의:

LLM API 장애가 있어도 Rule 기반 채점 결과는 반환해야 한다.

⸻

Phase 3: 피드백 생성 고도화

목표:

채점 결과를 바탕으로 자연어 피드백과 시험형 답안을 생성한다.

구현 범위:

* FeedbackGenerator
* 인정/누락 키워드 기반 피드백
* 시험형 추천 답안
* 한 줄 암기 포인트

⸻

Phase 4: 풀이 이력 저장 및 분석

목표:

사용자의 누적 답안 데이터를 기반으로 취약 영역을 분석한다.

구현 범위:

* UserAnswer 저장
* ScoringResult 저장
* 카테고리별 평균 점수
* 자주 누락한 키워드
* 최근 풀이 추이

⸻

Phase 5: 문제/채점기준 관리자 기능

목표:

문제와 채점 기준을 직접 관리할 수 있게 한다.

구현 범위:

* 문제 등록
* 문제 수정
* 채점 기준 등록
* 유사 표현 등록
* 배점 수정
* 문제 비활성화

초기에는 관리자 UI 없이 API나 seed data로 시작해도 된다.

⸻

Phase 6: 과목 확장

목표:

애플리케이션 보안에서 네트워크 보안, 시스템 보안 등으로 확장한다.

확장 순서:

1. 네트워크 보안
2. 시스템 보안
3. 정보보안 일반
4. 정보보안관리 및 법규

⸻

13. 데이터 구축 전략

초기에는 많은 문제보다 품질 좋은 문제/채점 기준이 중요하다.

초기 데이터 목표

* 애플리케이션 보안 문제 20~30개
* 문제당 채점 기준 3~6개
* 기준당 alias 3~10개
* 문제당 추천 답안 1개 이상
* 문제당 해설 1개 이상

문제 데이터 예시

문제:

SQL Injection 공격 원리와 대응 방안을 설명하시오. [6점]

채점 기준:

* 공격 원리: 입력값에 악의적인 SQL 삽입: 2점
* PreparedStatement/바인딩 변수 사용: 2점
* 입력값 검증: 1점
* DB 권한 최소화 또는 에러 메시지 노출 제한: 1점

추천 답안:

SQL Injection은 사용자 입력값에 악의적인 SQL 구문을 삽입하여 데이터베이스를 비정상적으로 조회, 수정, 삭제하는 공격이다. 대응 방안으로는 PreparedStatement 또는 바인딩 변수 사용, 화이트리스트 기반 입력값 검증, DB 계정 권한 최소화, 에러 메시지 노출 제한 등이 있다.

14. 주의사항

1. AI API 래퍼처럼 보이면 안 된다

단순히 사용자의 답안을 AI에게 보내고 “채점해줘”라고 요청하는 구조는 피해야 한다.

반드시 서비스 내부에 다음 구조가 있어야 한다.

* 문제 데이터
* 채점 기준 데이터
* 키워드/유사 표현 데이터
* 서버 점수 계산 로직
* 채점 결과 저장
* 피드백 생성

⸻

2. 최종 점수는 서버가 계산해야 한다

AI가 직접 5점, 6점 등 최종 점수를 결정하게 하면 결과 일관성이 떨어진다.

AI는 다음 정도만 판단한다.

* 이 답안이 특정 기준을 충족하는가?
* 부분적으로 충족하는가?
* 어떤 이유로 그렇게 판단했는가?

점수 부여는 서버의 ScoreCalculator가 수행한다.

⸻

3. LLM 응답은 항상 실패 가능성을 고려한다

LLM API는 다음 문제가 생길 수 있다.

* 응답 지연
* 응답 실패
* JSON 파싱 실패
* 잘못된 형식 응답
* 과도하게 관대한 판단
* 과도하게 엄격한 판단
* 환각

따라서 다음 처리가 필요하다.

* timeout 설정
* retry 제한
* JSON schema 검증
* fallback 처리
* confidence 기준 미달 시 UNKNOWN 처리
* Rule 기반 결과 우선 적용

⸻

4. 법규 과목은 초기 MVP에서 제외한다

정보보안관리 및 법규는 최신 법령, 개정사항, 공식 용어 관리가 중요하다.
초기 MVP에서 법규까지 포함하면 정확성 부담이 커진다.

법규는 추후 다음 조건이 갖춰진 뒤 확장한다.

* 공식 출처 기반 데이터 관리
* 법령 개정 이력 관리
* 답안 근거 링크 관리
* 개정 전/후 용어 구분

⸻

5. 문제/정답 데이터 저작권에 주의한다

기출문제, 해설, 교재 내용을 그대로 복제하면 저작권 문제가 발생할 수 있다.

초기 데이터는 다음 방식이 안전하다.

* 직접 만든 유사 문제 사용
* 공개 가능한 범위의 개념형 문제 사용
* 문제 문장을 새로 작성
* 해설과 추천 답안 직접 작성
* 출처가 필요한 경우 명확히 관리

⸻

6. 채점 결과를 확정적으로 표현하지 않는다

서비스는 실제 시험 채점자가 아니므로 “실제 점수”가 아니라 “예상 점수” 또는 “채점 기준 기반 예상 점수”로 표현한다.

권장 표현:

* 예상 점수
* 기준 기반 채점 결과
* 부분점수 가능성
* 보완 추천 답안

피해야 할 표현:

* 실제 시험 점수
* 합격 보장
* 반드시 정답
* 무조건 감점 없음

⸻

7. 과도한 기능 확장을 피한다

초기부터 다음 기능을 모두 넣으려 하면 MVP가 흐려진다.

* RAG
* 벡터DB
* 모의고사
* 랭킹
* 커뮤니티
* 결제
* 모바일 앱
* 전과목 대응

초기에는 채점엔진 품질이 가장 중요하다.

16. 포트폴리오 어필 포인트

이 프로젝트는 단순 AI API 연동 프로젝트가 아니라 다음 포인트를 강조해야 한다.

* 도메인 특화 채점 기준 모델링
* Rule + LLM 하이브리드 채점엔진 설계
* AI 응답을 그대로 신뢰하지 않고 서버 로직으로 점수 계산
* 채점 기준, 유사 표현, 부분점수 기준을 데이터화
* LLM 실패 시 fallback 가능한 구조
* 사용자 답안 이력 기반 취약점 분석
* 특정 과목부터 시작해 전과목으로 확장 가능한 설계
* AI 기반 서비스에서 신뢰성과 일관성을 확보하려는 설계

이력서/면접 설명 문장 예시:

AI 응답을 그대로 사용하는 방식이 아니라, 정보보안기사 실기 답안의 채점 기준과 부분점수 규칙을 도메인 모델로 분리하고, Rule 기반 키워드 매칭과 LLM 의미판정을 결합한 하이브리드 채점엔진을 설계했습니다. 최종 점수는 서버에서 계산하고, LLM은 기준 충족 여부 판단과 피드백 생성에만 사용하여 채점 일관성과 설명 가능성을 확보했습니다.

17. 최종 방향성

이 프로젝트의 첫 번째 성공 기준은 “전과목을 지원하는 것”이 아니다.

첫 번째 성공 기준은 다음이다.

사용자가 애플리케이션 보안 주관식 답안을 입력했을 때,
왜 이 답안이 몇 점인지, 어떤 키워드가 인정되었는지, 무엇이 누락되었는지, 어떻게 고치면 더 좋은 답안이 되는지를 납득할 수 있어야 한다.

즉, MVP의 핵심은 다음이다.

* 좁은 범위
* 높은 채점 품질
* 명확한 채점 근거
* 일관된 점수 계산
* 시험형 답안 개선

전과목 확장, 모의고사, 학습 플랜, 법규 최신성 반영은 이 핵심이 안정화된 이후의 목표로 둔다.
