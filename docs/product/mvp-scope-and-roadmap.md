# MVP 범위와 로드맵

> 이 문서는 무엇을 언제 만드는지(범위, 우선순위, 데이터 전략)를 정의한다.
> 제품의 존재 이유는 [product-overview.md](product-overview.md), 설계는 [service-design.md](service-design.md) 참고.

## 1. 초기 MVP 범위

초기 MVP는 애플리케이션 보안 과목 단일 영역으로 제한한다.

### 초기 지원 주제 예시

- SQL Injection
- XSS
- CSRF
- 파일 업로드 취약점
- 인증과 인가
- 세션 하이재킹
- 세션 고정 공격
- 비밀번호 저장 정책
- 입력값 검증
- 출력값 인코딩
- 접근통제 취약점
- 디렉터리 리스팅
- 경로 조작 공격
- 쿠키 보안 속성
- HTTPS/TLS 적용 목적

### 초기 문제 수

초기에는 20~30문제 정도를 목표로 한다.
문제 수를 무리하게 늘리기보다, 각 문제에 대해 채점 기준과 피드백 품질을 높이는 것이 우선이다.

### MVP에서 제공할 핵심 기능

- 문제 목록 조회
- 문제 상세 조회
- 사용자 답안 제출
- 채점 결과 반환
- 예상 점수 표시
- 인정 키워드 표시
- 누락 키워드 표시
- 부분 인정 항목 표시
- 감점 사유 표시
- 시험형 추천 답안 제공
- 풀이 결과 저장
- 간단한 카테고리/키워드별 취약점 분석

## 2. MVP에서 제외할 기능

초기 MVP에서는 다음 기능을 우선순위에서 제외한다.

- 전과목 대응
- 완성형 모의고사 시스템
- 랭킹/커뮤니티
- 결제
- 관리자용 대시보드 고도화
- 법규 최신성 자동 추적
- 복잡한 RAG 기반 지식 검색
- 대규모 벡터DB 기반 추천 시스템
- 모바일 앱
- 실시간 채팅 튜터

초기 목표는 "정보보안기사 전체 학습 플랫폼"이 아니라, **애플리케이션 보안 주관식 답안을 안정적으로 채점하는 엔진**을 만드는 것이다.

## 3. 과목 타겟팅 전략

### 특정 과목 타겟팅을 선택하는 이유

초기 MVP는 전과목 대응보다 특정 과목 타겟팅이 적합하다.

- 채점 기준 데이터를 빠르게 구축할 수 있다.
- 서비스 메시지가 명확하다.
- 채점엔진 품질을 빠르게 검증할 수 있다.
- 문제별 정답 키워드와 유사 표현 관리가 가능하다.
- 포트폴리오 관점에서 "도메인 특화 AI 채점엔진"으로 설명하기 좋다.
- 전과목 대응 시 발생하는 품질 편차를 줄일 수 있다.

### 애플리케이션 보안을 먼저 선택하는 이유

- 개발자 입장에서 도메인 이해가 쉽다.
- SQL Injection, XSS, CSRF 등 답안 표현이 다양하다.
- 정답 키워드와 유사 표현 관리 가치가 크다.
- AI 의미판정의 효용이 잘 드러난다.
- 부분점수 채점과 시험형 답안 변환 기능을 보여주기 좋다.
- 백엔드 개발자 포트폴리오와 연결성이 높다.

### 추후 확장 순서

1. 애플리케이션 보안
2. 네트워크 보안
3. 시스템 보안
4. 정보보안 일반
5. 정보보안관리 및 법규

법규 과목은 최신성, 용어 변경, 법령 근거 관리가 필요하므로 초기 MVP에서는 제외하고 후순위로 둔다.

## 4. 개발 우선순위

### Phase 1: 단일 문제 채점 MVP

**목표**: 한 문제에 대해 사용자가 답안을 제출하면 점수와 피드백을 받을 수 있다.

구현 범위:

- Problem 조회
- Rubric 조회
- 답안 제출
- Rule 기반 키워드 매칭
- 점수 계산
- 채점 결과 반환
- 기본 추천 답안 반환

이 단계에서는 LLM 없이도 동작 가능해야 한다.

### Phase 2: Rule + LLM 하이브리드 채점

**목표**: 단순 키워드 매칭으로 놓치는 유사 표현을 AI가 보조 판단한다.

구현 범위:

- LLM Client
- LlmCriterionEvaluator
- JSON 응답 파싱
- confidence 처리
- UNKNOWN 상태 처리
- LLM 실패 시 fallback 처리

주의: LLM API 장애가 있어도 Rule 기반 채점 결과는 반환해야 한다.

### Phase 3: 피드백 생성 고도화

**목표**: 채점 결과를 바탕으로 자연어 피드백과 시험형 답안을 생성한다.

구현 범위:

- FeedbackGenerator
- 인정/누락 키워드 기반 피드백
- 시험형 추천 답안
- 한 줄 암기 포인트

### Phase 4: 풀이 이력 저장 및 분석

**목표**: 사용자의 누적 답안 데이터를 기반으로 취약 영역을 분석한다.

구현 범위:

- UserAnswer 저장
- ScoringResult 저장
- 카테고리별 평균 점수
- 자주 누락한 키워드
- 최근 풀이 추이

### Phase 5: 문제/채점기준 관리자 기능

**목표**: 문제와 채점 기준을 직접 관리할 수 있게 한다.

구현 범위:

- 문제 등록
- 문제 수정
- 채점 기준 등록
- 유사 표현 등록
- 배점 수정
- 문제 비활성화

초기에는 관리자 UI 없이 API나 seed data로 시작해도 된다.

### Phase 6: 과목 확장

**목표**: 애플리케이션 보안에서 네트워크 보안, 시스템 보안 등으로 확장한다.

확장 순서:

1. 네트워크 보안
2. 시스템 보안
3. 정보보안 일반
4. 정보보안관리 및 법규

## 5. 데이터 구축 전략

초기에는 많은 문제보다 품질 좋은 문제/채점 기준이 중요하다.

### 초기 데이터 목표

- 애플리케이션 보안 문제 20~30개
- 문제당 채점 기준 3~6개
- 기준당 alias 3~10개
- 문제당 추천 답안 1개 이상
- 문제당 해설 1개 이상

### 문제 데이터 예시

문제:

> SQL Injection 공격 원리와 대응 방안을 설명하시오. [6점]

채점 기준:

- 공격 원리: 입력값에 악의적인 SQL 삽입: 2점
- PreparedStatement/바인딩 변수 사용: 2점
- 입력값 검증: 1점
- DB 권한 최소화 또는 에러 메시지 노출 제한: 1점

추천 답안:

> SQL Injection은 사용자 입력값에 악의적인 SQL 구문을 삽입하여 데이터베이스를 비정상적으로 조회, 수정, 삭제하는 공격이다. 대응 방안으로는 PreparedStatement 또는 바인딩 변수 사용, 화이트리스트 기반 입력값 검증, DB 계정 권한 최소화, 에러 메시지 노출 제한 등이 있다.
