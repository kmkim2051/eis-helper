# 개발 원칙과 주의사항

> 이 문서는 개발 전 과정에서 지켜야 할 원칙, 기술 스택, 포트폴리오 어필 포인트를 정의한다.
> 설계 상세는 [service-design.md](service-design.md) 참고.

## 1. 주의사항

### AI API 래퍼처럼 보이면 안 된다

단순히 사용자의 답안을 AI에게 보내고 "채점해줘"라고 요청하는 구조는 피해야 한다.

반드시 서비스 내부에 다음 구조가 있어야 한다.

- 문제 데이터
- 채점 기준 데이터
- 키워드/유사 표현 데이터
- 서버 점수 계산 로직
- 채점 결과 저장
- 피드백 생성

### 최종 점수는 서버가 계산해야 한다

AI가 직접 5점, 6점 등 최종 점수를 결정하게 하면 결과 일관성이 떨어진다.

AI는 다음 정도만 판단한다.

- 이 답안이 특정 기준을 충족하는가?
- 부분적으로 충족하는가?
- 어떤 이유로 그렇게 판단했는가?

점수 부여는 서버의 ScoreCalculator가 수행한다.

### LLM 응답은 항상 실패 가능성을 고려한다

LLM API는 다음 문제가 생길 수 있다.

- 응답 지연
- 응답 실패
- JSON 파싱 실패
- 잘못된 형식 응답
- 과도하게 관대한 판단
- 과도하게 엄격한 판단
- 환각

따라서 다음 처리가 필요하다.

- timeout 설정
- retry 제한
- JSON schema 검증
- fallback 처리
- confidence 기준 미달 시 UNKNOWN 처리
- Rule 기반 결과 우선 적용

### 법규 과목은 초기 MVP에서 제외한다

정보보안관리 및 법규는 최신 법령, 개정사항, 공식 용어 관리가 중요하다.
초기 MVP에서 법규까지 포함하면 정확성 부담이 커진다.

법규는 추후 다음 조건이 갖춰진 뒤 확장한다.

- 공식 출처 기반 데이터 관리
- 법령 개정 이력 관리
- 답안 근거 링크 관리
- 개정 전/후 용어 구분

### 문제/정답 데이터 저작권에 주의한다

기출문제, 해설, 교재 내용을 그대로 복제하면 저작권 문제가 발생할 수 있다.

초기 데이터는 다음 방식이 안전하다.

- 직접 만든 유사 문제 사용
- 공개 가능한 범위의 개념형 문제 사용
- 문제 문장을 새로 작성
- 해설과 추천 답안 직접 작성
- 출처가 필요한 경우 명확히 관리

### 채점 결과를 확정적으로 표현하지 않는다

서비스는 실제 시험 채점자가 아니므로 "실제 점수"가 아니라 "예상 점수" 또는 "채점 기준 기반 예상 점수"로 표현한다.

권장 표현: 예상 점수, 기준 기반 채점 결과, 부분점수 가능성, 보완 추천 답안

피해야 할 표현: 실제 시험 점수, 합격 보장, 반드시 정답, 무조건 감점 없음

### 과도한 기능 확장을 피한다

초기부터 다음 기능을 모두 넣으려 하면 MVP가 흐려진다.

- RAG
- 벡터DB
- 모의고사
- 랭킹
- 커뮤니티
- 결제
- 모바일 앱
- 전과목 대응

초기에는 채점엔진 품질이 가장 중요하다.

## 2. 추천 기술 구성

백엔드 개발자 포트폴리오 관점에서는 다음 구성이 적합하다.

### Backend

- Java 17 또는 21
- Spring Boot
- JPA
- MariaDB 또는 PostgreSQL
- Redis 선택사항
- OpenAI/Gemini 등 LLM API 연동

### Frontend

- React 또는 Next.js
- 문제 풀이 화면
- 채점 결과 화면
- 풀이 이력 화면
- 취약점 분석 화면

### Infra

- Docker Compose
- CI/CD 선택사항
- 초기에는 단순 배포로 충분
- 추후 Prometheus/Grafana 등 모니터링 확장 가능

### Optional

- Vector DB는 초기 MVP에서는 제외 가능
- RAG는 법규/해설 확장 단계에서 고려
- 관리자 페이지는 후순위

## 3. 포트폴리오 어필 포인트

이 프로젝트는 단순 AI API 연동 프로젝트가 아니라 다음 포인트를 강조해야 한다.

- 도메인 특화 채점 기준 모델링
- Rule + LLM 하이브리드 채점엔진 설계
- AI 응답을 그대로 신뢰하지 않고 서버 로직으로 점수 계산
- 채점 기준, 유사 표현, 부분점수 기준을 데이터화
- LLM 실패 시 fallback 가능한 구조
- 사용자 답안 이력 기반 취약점 분석
- 특정 과목부터 시작해 전과목으로 확장 가능한 설계
- AI 기반 서비스에서 신뢰성과 일관성을 확보하려는 설계

이력서/면접 설명 문장 예시:

> AI 응답을 그대로 사용하는 방식이 아니라, 정보보안기사 실기 답안의 채점 기준과 부분점수 규칙을 도메인 모델로 분리하고, Rule 기반 키워드 매칭과 LLM 의미판정을 결합한 하이브리드 채점엔진을 설계했습니다. 최종 점수는 서버에서 계산하고, LLM은 기준 충족 여부 판단과 피드백 생성에만 사용하여 채점 일관성과 설명 가능성을 확보했습니다.
