# API 설계 점검

대상: 인자가 있으면 해당 Controller/도메인, 없으면 프로젝트의 모든 `@RestController`와 관련 DTO.

## 점검 체크리스트

### URL과 HTTP 메서드
- [ ] URL은 명사 복수형 리소스인가 (`/api/users`, `/api/users/{id}`) — 동사 URL(`/getUser`, `/createUser`) 금지
- [ ] HTTP 메서드가 의미에 맞는가: 조회 GET, 생성 POST, 전체수정 PUT, 부분수정 PATCH, 삭제 DELETE
- [ ] 계층 표현이 자연스러운가 (`/users/{id}/orders`) — 3단계 넘게 깊어지면 지적
- [ ] 공통 prefix(`/api`)가 일관적인가

### 상태 코드와 응답
- [ ] 생성은 201, 삭제/본문 없는 성공은 204를 쓰는가 (전부 200이면 지적)
- [ ] 프로젝트 공통 응답 포맷이 있다면 모든 API가 따르는가; 없다면 도입 필요 여부를 한 번만 제안
- [ ] 에러 응답에 클라이언트가 분기할 수 있는 정보(코드/메시지)가 있는가

### DTO
- [ ] 엔티티를 요청/응답에 직접 노출하는가 (**가장 중요 — 반드시 잡는다**)
- [ ] 요청 DTO와 응답 DTO가 분리되어 있는가 (하나로 겸용하면 지적)
- [ ] 요청 DTO에 Bean Validation(`@Valid` + `@NotNull` 등)이 적용되어 있는가
- [ ] DTO는 record 또는 불변 클래스인가 (Java 21이므로 record 우선)

### 예외 처리
- [ ] `@RestControllerAdvice` 전역 핸들러가 있는가; Controller마다 try-catch로 처리하면 지적
- [ ] 도메인 예외 → HTTP 상태코드 매핑이 일관적인가 (없는 리소스 404, 검증 실패 400, 권한 403)

### Controller 형태
- [ ] Controller 메서드가 위임 이상의 일을 하는가 (비즈니스 로직은 /oop-check 영역이지만 눈에 띄면 함께 지적)
- [ ] 페이징이 필요한 목록 API에 `Pageable`/페이징 응답이 있는가

## 출력 형식

먼저 현재 API 목록을 표로 요약한다: `메서드 | URL | 설명`.
그 다음 발견 사항을 심각한 순으로:

```
N. [파일:라인] 문제 한 줄 요약
   - 제안: 구체적인 수정 방향 (필요하면 시그니처/URL 예시 포함)
```

발견이 없으면 "점검 통과"와 확인한 Controller 목록을 알린다.

## 규칙

- **제안만 하고 코드를 생성하거나 수정하지 않는다.** 사용자가 지시하면 그때 생성하거나 수정한다.
- 아직 Controller가 없으면, 점검 대신 이 체크리스트를 기준으로 한 API 설계 가이드를 짧게 안내한다.
- 새로운 컨벤션(공통 응답 포맷 등)을 제안해서 채택되면, 이 SKILL.md의 체크리스트에 반영하라고 사용자에게 알린다.
