# OOP 리팩토링 점검

대상: 인자가 있으면 해당 패키지/클래스, 없으면 `git diff`(스테이지 포함)의 변경 파일. 변경이 없으면 최근 커밋 1~2개의 변경 파일.

## 점검 체크리스트

### 책임과 크기
- [ ] 한 클래스가 여러 이유로 변경되는가 (SRP 위반, 갓 클래스)
- [ ] Service가 지나치게 비대한가 — 도메인 로직이 엔티티/도메인 객체로 갈 수 있는데 Service에 절차형으로 쌓여 있는가
- [ ] 메서드가 30줄을 크게 넘거나, 추상화 수준이 섞여 있는가 (고수준 흐름과 저수준 조작이 한 메서드에)

### 레이어 규칙 (Controller → Service → Repository)
- [ ] Controller에 비즈니스 로직이 있는가 (Controller는 요청 파싱, 위임, 응답 변환만)
- [ ] Repository를 Controller에서 직접 호출하는가
- [ ] `@Transactional` 경계가 Service에 있는가 (Controller나 Repository에 있으면 지적)

### 도메인 모델
- [ ] 엔티티가 getter/setter만 있는 데이터 덩어리인가 — 상태 변경 로직이 엔티티 메서드로 들어갈 수 있는지
- [ ] 무분별한 `@Setter` 사용 — 생성자/정적 팩토리/의도가 드러나는 메서드로 대체 가능한지
- [ ] `@Data`를 엔티티에 쓰는가 (equals/hashCode, toString 문제 — 금지)

### 결합도와 의존성
- [ ] 필드 주입(`@Autowired` 필드)이 있는가 — 생성자 주입(`@RequiredArgsConstructor`)으로
- [ ] 구체 클래스가 아닌 역할(인터페이스)에 의존해야 할 지점인데 구체 타입에 묶여 있는가 (단, 구현체가 하나뿐이면 인터페이스 강요하지 않음)
- [ ] 순환 참조 (Service 간 상호 의존)

### 중복과 일관성
- [ ] 같은 검증/변환 로직이 여러 곳에 복붙되어 있는가
- [ ] 기존 코드에 이미 있는 유틸/패턴을 안 쓰고 새로 만들었는가

## 출력 형식

발견 사항을 심각한 순으로 번호 목록으로 정리한다. 각 항목:

```
N. [파일:라인] 문제 한 줄 요약
   - 왜 문제인지 (한 줄)
   - 제안: 구체적인 리팩토링 방향
```

발견이 없으면 "점검 통과"와 확인한 범위를 짧게 알린다.

## 규칙

- **제안만 하고 코드를 수정하지 않는다.** 사용자가 "고쳐줘" 또는 번호를 지정하면 그때 수정한다.
- 이론적으로 가능한 지적이 아니라, 이 코드베이스에서 실제로 문제가 될 것만 올린다. 억지 지적으로 목록을 채우지 않는다.
- 초기 단계 프로젝트임을 감안해 과도한 추상화(불필요한 인터페이스, 이른 일반화)를 오히려 지적한다.
